# EMR Summarization & Evaluation Pipeline

Решение задачи хакатона **Арена LLM** — «Разработка методов генерации и автоматизированной оценки качества суммаризации медицинских текстов».

Задача: по тексту электронной медицинской карты (ЭМК) пациента, направленного на **КТ органов брюшной полости**, автоматически сформировать структурированное клиническое резюме и оценить его качество.

---

## Архитектурные решения

### Локальная LLM вместо внешнего API

Медицинские данные ЭМК — специальная категория персональных данных (ФЗ-152). Передача таких данных во внешние API-сервисы юридически недопустима. Весь пайплайн работает локально.

Модель: **ai-sage/GigaChat3.1-10B-A1.8B-GGUF** (квантизация q6_K, лицензия MIT).  
Запуск: MacBook Pro M1 Pro, 16 ГБ RAM — весь пайплайн выполняется на CPU/Apple Metal.

### Directed Generation через Outlines

Библиотека [Outlines](https://github.com/outlines-dev/outlines) интегрируется на уровне логитов модели и физически ограничивает пространство возможных токенов в соответствии с заданной JSON Schema. Это означает, что модель **не может** вернуть ответ, не соответствующий схеме: неправильный формат, пропущенные обязательные поля и недопустимые типы данных исключены на уровне генерации, а не постобработки.

### LangGraph как оркестратор

Оба пайплайна реализованы в виде LangGraph-графов. Это создаёт готовый фундамент для перехода к **Agentic Workflow**: объединения суммаризатора и оценщика в единый агентский контур с обратной связью (Loop), где оценщик становится итеративным корректором.

## Файлы репозитория

```
.
├── summarization.py   # Пайплайн суммаризации (схемы + логика + CLI)
└── evaluation.py      # Пайплайн оценки качества (схемы + логика + CLI)
```

### `summarization.py`

Самодостаточный пайплайн суммаризации ЭМК. Содержит Pydantic-схемы, системные промпты, узлы LangGraph и CLI.

Схема графа:

```
START → SafetyExtractor → AnatomyClinicalExtractor → OncoExtractor → LabImgExtractor → SummaryComposer → END
```

Узлы 1–4 последовательно извлекают структурированные JSON-объекты из исходного ЭМК через directed generation (Outlines + Pydantic). Узел 5 (SummaryComposer) получает на вход **агрегат JSON** от четырёх предыдущих узлов и генерирует финальное текстовое резюме по заданному шаблону. Демографические данные (возраст, пол) извлекаются отдельно через RegEx из заголовка ЭМК.

Что извлекается:
- **Node 1 — SafetyData**: аллергии на контраст/йод, функция почек (креатинин, СКФ), нефротоксичные препараты, антикоагулянты, метформин
- **Node 2 — AnatomyClinicalData**: жалобы, хронология заболевания, операции на ОБП, сопутствующая патология
- **Node 3 — OncoData**: онкологический анамнез — каждый диагноз как отдельная запись (локализация, стадия TNM, лечение, ответ на терапию)
- **Node 4 — LabImgData**: лабораторные показатели и инструментальные находки с числовыми значениями, единицами и датами
- **Node 5 — SummaryComposer**: финальное текстовое резюме по клиническому шаблону (свободная генерация)

Запуск:

```bash
python summarization.py path/to/emr.txt

# с указанием выходного файла:
python summarization.py path/to/emr.txt --out path/to/summary.txt

# с подробными логами llama.cpp:
python summarization.py path/to/emr.txt --debug
```

Результат: текстовый файл резюме + debug JSON со всеми извлечёнными структурированными данными.

### `evaluation.py`

Самодостаточный пайплайн оценки качества резюме (LLM-as-judge). Содержит Pydantic-схемы, системный промпт, узел LangGraph и CLI.

Схема графа:

```
START → EvaluatorNode → END
```

Единственный узел получает пару (исходный ЭМК, резюме), генерирует `EvalReport` через Outlines (directed generation по JSON Schema), вычисляет баллы по 5 критериям и сохраняет текстовый отчёт.

Принцип оценки — **Zero-Tolerance**: каждая из 20 бинарных проверок получает 1 только если факт присутствует, абсолютно точен, не обобщён и не искажён. Частичное совпадение = 0.

Критерии оценки:
- **Критерий 1 — Безопасность** (5 проверок): аллергия, креатинин, СКФ, нефрологические диагнозы, препараты риска
- **Критерий 2 — Полнота охвата** (4 проверки): жалобы, анамнез, онкологический статус, лаб/инструментальные данные
- **Критерий 3 — Фактическая точность** (4 проверки): числовые данные, диагнозы, галлюцинации, интерпретация
- **Критерий 4 — Структура и формат** (3 проверки): наличие разделов, лаконичность, достаточный объём
- **Критерий 5 — Клиническая значимость** (4 проверки): срочность КТ, онко-нефрологический контекст, информационный шум, actionable-рекомендации

Максимальный балл: **20 / 20**. Для каждого критерия модель формирует поле `rationale` с кратким описанием нарушений.

Запуск:

```bash
python evaluation.py --source_file path/to/emr.txt --summary_file path/to/summary.txt

# с указанием выходного файла:
python evaluation.py --source_file emr.txt --summary_file summary.txt --out report.txt

# с подробными логами llama.cpp:
python evaluation.py --source_file emr.txt --summary_file summary.txt --debug
```

Результат: текстовый отчёт с баллами по каждому критерию, rationale по каждому нарушению и итоговой оценкой.

Шкала итоговой оценки:

| Балл | Уровень |
|------|---------|
| 19–20 | Отлично — идеальная точность |
| 16–18 | Хорошо — незначительные потери специфичности |
| 12–15 | Удовлетворительно — присутствуют критичные пропуски |
| < 12  | Неудовлетворительно — опасная потеря данных |

## Установка зависимостей

```bash
pip install outlines==1.2.12 llama-cpp-python langgraph pydantic
```

Для Apple Silicon (Metal-ускорение):

```bash
CMAKE_ARGS="-DLLAMA_METAL=on" pip install llama-cpp-python --force-reinstall --no-cache-dir
```

Скачать модель:

```bash
# ai-sage/GigaChat3.1-10B-A1.8B-GGUF (лицензия MIT)
# https://huggingface.co/ai-sage/GigaChat3.1-10B-A1.8B-GGUF
```

После загрузки укажи путь к `.gguf`-файлу в переменной `MODEL_PATH` в начале каждого скрипта.

## Адаптация под другую задачу

Оба файла самодостаточны и содержат схемы прямо внутри — адаптация под другую доменную область:

1. Замени Pydantic-схемы на нужные сущности и проверки.
2. Обнови системные промпты под свою область и метрику качества.
3. Замени `_format_prompt()` под chat-template своей модели (Llama 3, Mistral, Qwen и др.).
4. Скорректируй `CRITERION_CHECKS` в `evaluation.py` если изменился набор проверок.
5. Подбери лимиты контекста (`MAX_CTX`, `MAX_INPUT`) под размер своих документов.
